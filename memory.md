Once the user added `GEMINI_API_KEY`, every future capture risks the whole function being killed by the platform timeout mid-flight, which the client only ever sees as "no data came back," not a real error. Not yet confirmed against production (cannot reach `*.vercel.app` from this environment, per the standing known issue) — **ask the user to do a fresh real capture after this deploys and report whether the title/summary come through correctly now.** If it still fails, the next diagnostic step is the actual Vercel function logs (Dashboard → Logs/Functions), which would show the `console.error("extract metadata failed", error)` line with the real underlying error — something this agent cannot retrieve itself.
- **THE ACTUAL ROOT CAUSE (found via Vercel function logs, which the user fetched and pasted in — this is the right way to diagnose anything Vercel-runtime-specific that this agent cannot reach directly)**: `/api/extract` was crashing on **every single invocation** in production with `Error [ERR_MODULE_NOT_FOUND]: Cannot find module '/var/task/src/utils/extraction.ts'`. The maxDuration fix above was a red herring — the function never got far enough to time out, it crashed on import before the handler even ran. Cause: `api/extract.ts` imported `from "../src/utils/extraction.ts"` and `src/utils/extraction.ts` imported `from "../data/types.ts"` — **explicit `.ts` extensions on relative imports**. Local Vite dev resolves this fine (and in fact `tsconfig.node.json`, which type-checks `vite.config.ts`'s import graph, uses `"module": "nodenext"`, which *requires* an explicit extension — this is *why* the `.ts` extensions were there in the first place). But Vercel's serverless function bundler does not rewrite that extension when transpiling, so the deployed function tries to import a literal `.../extraction.ts` path that doesn't exist post-build, and crashes immediately — explaining why extraction *always* silently fell back to hostname-only drafts in production, completely independent of the network/timeout/Gemini theories above. **Fix**: changed the extensions from `.ts` to `.js` (not removed entirely) — this is TypeScript's documented NodeNext convention: write `.js`, the compiler maps it to the `.ts` source for type-checking, and it's also the correct extension post-compilation, satisfying both `tsconfig.node.json`'s nodenext strictness *and* Vercel's runtime module resolution simultaneously. Verified locally (extraction still returns real titles/content) and via clean `tsc -b`/`npm run build` before pushing (`4497030`). **If any other current or future file under `api/` imports a sibling file with an explicit `.ts` extension, apply the same `.ts` → `.js` fix** — `api/agent.ts` was already written without extensions at all (works fine under `tsconfig.app.json`'s `"bundler"` resolution, just not reachable from `vite.config.ts`'s nodenext-resolved import graph, which is why it never hit this bug).
- **Process note for future sessions**: when something can only fail in the real deployed environment and this agent's tools cannot reach `*.vercel.app`, the fastest real diagnostic is **asking the user to open Vercel → Logs (or Deployments → a specific deployment → Functions) and paste the actual error**, not guessing further from source review alone. Two prior fixes this round (Gemini endpoint shape, maxDuration) were real, legitimate bugs worth fixing — but neither was *the* bug causing the symptom the user was actually seeing. The logs resolved it in one step.

## Decisions

- **Polling (3s interval), not Supabase Realtime, for cross-device sync.** Originally implemented via `postgres_changes` + a Realtime channel subscription. Debugged extensively with the user directly in the Supabase dashboard (confirmed: table correctly in the `supabase_realtime` publication, INSERT/UPDATE/DELETE all enabled, RLS open) — the channel reached `SUBSCRIBED` status but **no postgres_changes events ever arrived**, even when testing via Supabase's own Realtime Inspector. Root cause not resolved (likely something dashboard/project-config-specific not visible to either of us). Given the hackathon time pressure, made the deliberate call to rip it out and use simple polling instead: `setInterval(refetch, 3000)` plus an immediate `refetch()` after every local mutation for instant same-tab feedback. Verified working end-to-end including a true cross-device simulation (inserted a row via raw `fetch` POST to the Supabase REST API, confirmed the running app picked it up via polling alone within 3s, with zero action taken in that tab). This is the "boring solution that works" — slightly higher latency than true Realtime, but robust and fully understood, which matters more for a live demo than shaving 2-3 seconds.
- **No auth, open RLS.** Confirmed with user — single shared dataset for a one-day demo. Must not carry forward into any real/non-demo use without adding auth + scoped RLS.
- **`id` column is `text`, not `uuid`.** Needed because seed entries use ids like `"e1"`–`"e7"`; new entries get a server-generated UUID string via `default gen_random_uuid()::text`, same column.
- Decisions carried over from the mock-MVP build (still valid): single SPA over 4 static pages (for the closed-loop experience), 不留了 is a real delete not soft-delete, 3-column workbench uses spacing/typography not borders (avoids kanban look), simple chip-based project picker.
- **Investigated but did not fix the Realtime issue further**: compared against the user's other project (`lumihelia/housewarming-guestbook`, a single-file Alpine.js + Supabase site) which uses the *identical* `channel().on('postgres_changes', ...).subscribe()` pattern against a *different* Supabase project, and its Realtime works. Confirms the bug is specific to this project's Supabase instance/config, not our client code — but the actual root cause in that instance was never found. Not worth re-investigating unless a future session has time and wants true sub-second sync; polling is the accepted permanent-for-now solution.
- **Deployed via Vercel's web dashboard, not the CLI** — the CLI could not reach `api.vercel.com` from this machine's agent terminal (network-level connection resets), even with a personal access token. The dashboard "Import Git Repository" flow worked without issue.
- **Round 8 product decision: deepen the existing row model for today's demo, do not split into a real graph yet.** The richer conceptual model is real, but implementing separate Material/Thought/Claim/Relation tables before the 18:59 demo would create migration and UI risk. The current row model is a deliberate demo bridge; future work can split it once the core loop is proven.
- **Gemini is optional and server-side only.** `api/extract.ts` will call Gemini when `GEMINI_API_KEY` exists; otherwise it uses metadata/fallback extraction. Do not put Gemini keys in frontend env vars or source code.
